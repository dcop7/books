# Overview and high-level themes

Martin Kleppmann’s *Designing Data-Intensive Applications* is a systems architecture book focused on how to build applications that handle large volumes of data and operate reliably and maintainably under real-world conditions. The book is organized into three parts:

* **Foundations** — data models, storage engines, encoding/evolution.
* **Distributed Data** — replication, partitioning, transactions, and coordination.
* **Derived Data** — processing models: batch and streaming, and how to build derived views.

Across all parts the author emphasizes the big tradeoffs: consistency vs availability vs latency, simplicity vs features, correctness vs performance, and maintainability vs cleverness. Kleppmann often provides conceptual frameworks and concrete algorithms and then assesses tradeoffs so that architects can make principled design decisions.

Key takeaways:

* Different data models and query languages fit different tasks; no single model is best for everything.
* Storage engines (LSM trees vs B-trees) have different read/write tradeoffs; understanding these is essential when building systems.
* Distributed systems bring complexity: replication improves availability but introduces consistency challenges. Partitioning (sharding) helps scalability but complicates transactions and joins.
* Transactions provide safety and simplify reasoning but can be expensive or limited in distributed settings.
* The log (append-only sequence of records) is a fundamental abstraction; many systems build on logs (e.g., Kafka, databases with write-ahead logs).
* Derived data (materialized views, caches, analytics) needs careful handling to maintain correctness and freshness; batch and streaming processing are complementary.
* Coordination (consensus algorithms like Paxos/Raft) is necessary for some problems but expensive; minimizing coordination where possible is a useful design principle.

# Part I — Foundations of Data Systems

## Chapter 1 — Reliable, scalable, and maintainable applications

**Core ideas**

* **Reliability:** The system continues to operate correctly despite hardware/software failures, network issues, and human mistakes. Reliability includes correctness (no data corruption) and availability (serving requests).
* **Scalability:** The system can handle growth in data size, throughput, and number of users. Scaling can be vertical (bigger machines) or horizontal (add more machines).
* **Maintainability:** The system can be understood, modified, and extended by engineers. Maintainability includes operability (monitoring, debugging) and simplicity of design.

**Design goals and failure modes**

* Understand different failure modes: hardware failures, software bugs, operator error, data center outages, network partitions.
* Failures should be handled gracefully: detect, isolate, recover, and preferably automate recovery.
* Emphasize observable systems: metrics, logs, tracing. Observability helps diagnose issues and reason about system behaviour.

**Workload characteristics**

* **Read-heavy vs write-heavy** workloads affect architectural choices (caches, replication strategies).
* **OLTP vs OLAP**: transactional online workloads require low-latency and strong correctness guarantees; analytical workloads can tolerate high latency and favor throughput.

**Architectural patterns**

* Use of proxies, caches, queues, and stream processors to decouple system components and smooth spikes in load.
* Decompose systems into microservices or modular components: simpler to develop, independently deploy, but adds operational complexity and distributed coordination needs.

**Design heuristics and tradeoffs**

* Prefer simple models for maintainability unless complexity buys clear benefits.
* Measure and benchmark real workloads to validate choices.

## Chapter 2 — Data models and query languages

**Data models covered**

* **Relational model (tables, rows, SQL)** — mature, powerful, standardized. Suited for structured data and complex queries (joins, aggregations).
* **Document model (JSON/BSON)** — flexible schema, nested structures; good for hierarchical data and evolving schemas.
* **Graph model (nodes, edges)** — expressive for highly connected data (social graphs, recommendation systems).
* **Key-value model** — simplest: map from key to opaque value; excellent for caching and simple storage.

**Query languages**

* SQL: declarative, powerful optimizer, mature ecosystem; strong for ad-hoc queries and analytics.
* Document queries: may have rich query languages (MongoDB query DSL), but less expressive for complex joins.
* Graph query languages: e.g., Gremlin, Cypher; specialized for traversal operations.

**Schema vs schema-less**

* Tradeoff: flexibility (schema-less) vs. predictable semantics and queryability (schema). Use schema validation and evolution mechanisms if your model supports it.

**Joins and denormalization**

* Joins are expensive in distributed systems. Denormalization (duplicating data) improves read performance but increases complexity for updates.
* Materialized views and secondary indexes are alternatives to compute joins ahead of time.

**Indexes**

* Indexes speed up lookups but add write overhead and storage cost.
* Choose indexes according to query patterns; understanding the storage engine matters for index performance.

**When to choose which model**

* Start from access patterns: the model that makes reads and writes simpler and cheaper is usually best.
* Consider future evolution: if fields change often, a flexible format helps; if complex queries are needed, relational may be better.

## Chapter 3 — Storage and retrieval

**Primary storage engine concepts**

* Two dominant families: **Log-structured Merge (LSM) trees** and **B-trees**.

**B-trees**

* Widely used in relational DBs.
* Good for random reads and in-place updates.
* Pages (nodes) are modified in place leading to random writes; performance depends on disk seek latency (mitigated by SSDs).

**LSM trees**

* Designed for high write throughput: writes are appended to an in-memory structure (memtable) and periodically flushed/merged into SSTables on disk.
* Reads may involve checking multiple levels; bloom filters and compaction mitigate read overhead.
* Great for write-heavy workloads and SSDs.

**Storage considerations**

* Caching: data and index pages in memory speed up reads.
* Compression: reduces storage and IO cost at the expense of CPU.
* Compaction and space amplification: LSM compaction causes write amplification and needs tuning.

**File formats and layouts**

* Row-oriented vs column-oriented storage:

  * **Row stores** excel at retrieving whole rows (OLTP).
  * **Column stores** excel at aggregations over columns (OLAP).
* Columnar compression is often more effective.

**Indexes and secondary indexes**

* Secondary indexes enable lookup by non-primary key fields but require maintenance on updates. In distributed stores, maintaining consistent secondary indexes is challenging.

**Disk vs memory tradeoffs**

* Memory is scarce/expensive; disk/SSD has higher latency and persistence. Choose caching strategies accordingly.

## Chapter 4 — Encoding and evolution

**Data encoding**

* Binary encodings (Avro, Protocol Buffers, Thrift) vs textual (JSON, XML).
* Binary formats are more compact and efficient to parse; textual formats are easier to debug and evolve without special tools.

**Schema evolution**

* Schema changes are inevitable. Good systems support forward and backward compatibility to allow producers and consumers to evolve independently.
* Techniques:

  * **Versioned schemas** with schema registry (e.g., Avro + Schema Registry). Consumers and producers can negotiate schemas.
  * **Compatibility rules**: adding optional fields, making fields nullable, default values, avoiding incompatible changes.

**Data migration strategies**

* **Lazy migration** — keep old data on disk and convert on read.
* **Eager migration** — transform all stored data to new schema (expensive).
* **Dual writers / readers** — write in multiple formats during migration.

**APIs and formats**

* Design stable, well-documented APIs; prefer explicit versioning, deprecation policy.
* Use strong typing and validation for maintainability.

# Part II — Distributed Data

## Chapter 5 — Replication

**Why replicate?**

* Fault tolerance: keep copies of data in multiple machines to survive failures.
* Read scalability: serve reads from replicas.
* Geographic distribution: reduce latency by placing data near users.

**Replication models**

* **Single leader (master-replica)** — a leader handles writes; replicas serve reads asynchronously. Simple to reason about but leader is a single point for writes.
* **Multi-leader (master-master)** — multiple nodes accept writes; requires conflict resolution. Good for geo-distributed writes but risky for conflicting updates.
* **Leaderless (gossip/optimistic replication)** — writes can go to multiple nodes; read repair and quorum rules reconcile state (e.g., Dynamo-style systems).

**Consistency models**

* **Synchronous vs asynchronous replication** — synchronous ensures replicas agree before acknowledging writes (stronger consistency, higher latency); asynchronous improves latency but may lose acknowledged writes on failure.
* **Eventual consistency** — replicas converge over time; clients might see stale data temporarily.
* **Causal consistency, read-your-writes, monotonic reads** — weaker than linearizability but stronger than pure eventual consistency; many systems aim to provide such guarantees.

**Quorum systems**

* Use read and write quorums (R, W, N) to ensure that reads observe recent writes if `R + W > N`.
* Tuning R and W trades availability vs latency vs consistency.

**Replica lag and failover**

* Replica lag causes stale reads; detect and avoid serving reads from lagging replicas in some contexts.
* Failover needs careful handling of split-brain and data divergence. Consensus (e.g., Raft) can be used to elect leaders safely.

**Conflict resolution**

* **Last writer wins (timestamp-based)** — simple but can lose updates.
* **CRDTs (Conflict-free Replicated Data Types)** — mathematically designed to merge concurrent updates without data loss, at the cost of constraints on operations and data types.
* **Application-level reconciliation** — domain specific logic to merge conflicts.

**Tradeoffs and operational concerns**

* Monitor replication lag, network partitions, and consistency semantics.
* Consider read-only replicas for analytics; avoid long-running reads holding resources.

## Chapter 6 — Partitioning (Sharding)

**Why partition?**

* A way to split data across multiple machines to scale beyond a single machine’s capacity (storage, CPU, network).

**Partitioning strategies**

* **Horizontal partitioning (sharding)** by key range or hash:

  * Range partitioning preserves locality for ordered scans but suffers from hotspots.
  * Hash partitioning evens distribution but breaks locality.
* **Directory-based partitioning** — a mapping service tells where a key lives (good for flexible migration).
* **Composite partitioning** — multi-dimensional partitioning for complex workloads.

**Rebalancing and resharding**

* Moving partitions between nodes requires careful handling to avoid downtime and inconsistent routing. Techniques include:

  * Copy data to new nodes then cut over.
  * Use consistent hashing to minimize data movement.
  * Use a directory service to migrate partitions gradually.

**Impact on queries**

* Cross-partition queries (joins, aggregate across shards) are expensive. Strategies:

  * Send query to all relevant partitions and aggregate results.
  * Repartition data for common joins (co-partitioning), or denormalize.

**Partition tolerance vs transactional guarantees**

* Partitioning complicates transactions: distributed transactions across partitions require coordination (2-phase commit, Paxos/Raft) or eventual consistency with compensating actions.

**Hotspots and skews**

* Hot keys (popular items) can overload the single partition. Strategies: application-level splitting (fan-out), caching, or adaptive partitioning.

## Chapter 7 — Transactions

**ACID properties**

* **Atomicity, Consistency, Isolation, Durability** — foundation for transactional systems. Database transactions simplify application reasoning.

**Isolation levels**

* Serializability (strongest), snapshot isolation, repeatable reads, read committed, read uncommitted.
* Lower isolation levels permit higher concurrency and lower latency but can expose anomalies (lost updates, dirty reads, write skew).

**Concurrency control mechanisms**

* **Pessimistic locking** — acquire locks for writes; prevents conflicts but reduces concurrency and risks deadlocks.
* **Optimistic concurrency control (OCC)** — allow concurrent execution and check for conflicts at commit time (e.g., MVCC systems).
* **Multi-version Concurrency Control (MVCC)** — keep multiple versions to enable non-blocking reads and snapshot isolation.

**Distributed transactions**

* **Two-phase commit (2PC)** — coordinates commit across multiple nodes; blocks participants if coordinator fails.
* **Three-phase commit (3PC)** — adds another phase to reduce blocking, but still complex.
* **Consensus-based replication (Paxos/Raft)** — can be used to implement transactions with less blocking by serializing updates through a single leader or consensus log.

**Alternative designs to avoid distributed transactions**

* **Eventual consistency plus application-level compensation** — using idempotent operations, sagas (long-running transactions structured as sequence of compensating actions), and CRDTs.
* **Partitioning strategies** that keep transactional sets localized to a single shard (co-location) to avoid distributed transactions.

**Snapshot isolation and anomalies**

* Snapshot isolation reduces many concurrency problems but allows anomalies like write skew. Serializable isolation is the only level guaranteeing correctness as if transactions were executed serially.

**Design recommendations**

* Prefer single-partition, single-node transactions when possible.
* Use distributed transactions sparingly; understand costs and failure modes.
* Consider sagas for long-running workflows.

# Part III — Derived Data

## Chapter 8 — The trouble with distributed systems (coordination, consensus)

**Coordination problems**

* Coordination is necessary when multiple nodes must agree (leader election, metadata management, commits).
* Coordination is expensive (latency) and sometimes fragile (consensus algorithms are complex).

**Consensus algorithms**

* Paxos and Raft are the two canonical algorithms for achieving consensus on a set of nodes.
* **Raft** is designed to be understandable and often used in real systems (etcd, Consul).
* Consensus is used for leader election, metadata storage, and serializing updates.

**When to coordinate**

* Minimum necessary: avoid global coordination for operations that can be done independently.
* Use coordination only where it buys correctness guarantees that matter.

**Tradeoffs**

* Consensus provides strong guarantees but adds latency and operational complexity. Design systems to minimize the number of operations that require consensus.

## Chapter 9 — Batch processing (MapReduce, Hadoop)

**MapReduce model**

* Two phases: **map** (transform input to key/value pairs) and **reduce** (aggregate values for each key).
* Well suited for embarrassingly parallel data processing (ETL, batch analytics).

**Hadoop and its ecosystem**

* HDFS for distributed storage; Yarn for scheduling; MapReduce or higher-level tools (Pig, Hive) for processing.
* Batch processing is high throughput but high latency.

**Cost of batch processing**

* Job startup overhead, data shuffles, sorting and merging.
* Suitable for offline analytics, periodic ETL, and large scans.

**Optimizations**

* Partitioning and sorting data, map-side combiners, reduce-side aggregation.
* Columnar storage and compression to speed analytics.

**Alternatives and evolution**

* Spark introduced DAG-based execution, in-memory processing, and richer APIs for iterative and interactive analytics — bridging the gap between batch and streaming.

## Chapter 10 — Stream processing (event streaming, streaming systems)

**Streaming vs batch**

* Streaming: process records as they arrive (low latency).
* Micro-batch: small batches over time windows (e.g., early Spark Streaming).
* Continuous streaming: record-at-a-time processing with state (e.g., Flink, Kafka Streams).

**Event streams and the log abstraction**

* The ordered, append-only log (e.g., Kafka) is the backbone for many modern architectures. The log can be replayed to rebuild state or recompute derived data.
* Event sourcing: store every state change as an event; state is recomputed by replaying events.

**Stream processing models**

* Stateless vs stateful operators.
* Windowing: processing time vs event time; handling late data and watermarks.
* Exactly-once vs at-most-once vs at-least-once processing semantics.

  * Exactly-once semantics are difficult but achievable end-to-end with careful design (idempotency, transactional sinks).
  * Kafka + Kafka Streams or transactional message sinks attempt to provide exactly-once semantics, but subtle details matter.

**State management**

* Local state vs remote storage; checkpointing and fault recovery.
* Techniques for maintaining consistent state across failures: periodic snapshots, changelogs, and writes to durable logs.

**Joining streams and tables**

* Stream-table joins, stream-stream joins — require state and often windowing.
* The choice of window semantics affects correctness and resource use.

**Use cases**

* Real-time analytics, monitoring, alerts, materialized views, and event-driven architectures.

# Cross-cutting topics, design patterns, and tradeoffs

**Logs as a fundamental abstraction**

* The log is a universal primitive: write-ahead logs, Kafka topics, change data capture stream — they enable replayability, recovery, and decoupling producers from consumers.

**Materialized views and derived data**

* Derived data is a copy of processed primary data aimed at speeding queries. Maintain materialized views via batch jobs or stream processors. Keep consistency expectations explicit.

**Idempotency and retries**

* Design operations to be idempotent to ease retries in distributed systems. Use unique request IDs or logical deduplication.

**Backpressure and flow control**

* Downstream systems can be slower — need strategies to handle bursts (rate limiting, buffering, shedding load).

**Monitoring and observability**

* Collect metrics, logs, and distributed traces. Instrumentation is key to understanding system behaviour and diagnosing issues.

**Testing distributed systems**

* Chaos engineering (fault injection), unit tests, property-based testing for invariants, simulation of network partitions.

**Security and compliance**

* Data governance, encryption at rest/in transit, access control, auditing. Consider privacy and regulatory requirements when replicating and shipping data.

**Cost considerations**

* Each design decision (replication factor, retention policies, indexing) impacts infrastructure cost.

# Practical implementation notes and checklist

Use this checklist when designing or evaluating a data-intensive application:

1. **Characterize workloads**

   * Read/write ratio, request latency requirements, data size, query patterns.

2. **Choose data model**

   * Match model (relational/document/graph/key-value) to access patterns and evolve plan.

3. **Select storage engine**

   * If write heavy: LSM; if random reads/updates: B-tree; if analytical: columnar.

4. **Design partitioning and replication**

   * Plan partition key to minimize cross-shard operations; set replication strategy for fault tolerance and read scaling.

5. **Decide consistency model**

   * Which guarantees are necessary? Strong consistency, causal, eventual?

6. **Plan for schema evolution**

   * Use typed schemas, schema registry, or versioning policies.

7. **Transaction strategy**

   * Prefer single-partition transactions; use sagas for long workflows; if distributed transactions required, understand 2PC and failure modes.

8. **Processing model for derived data**

   * Use batch for large offline processing; streaming for low-latency continuous updates; consider hybrid approaches.

9. **Observability**

   * Instrument: metrics, logs, tracing; define SLOs/SLAs and alerting rules.

10. **Operational procedures**

    * Backups and restores, disaster recovery drills, capacity planning, runbooks.

11. **Security**

    * RBAC, encryption, auditing, data retention policies.

12. **Testing**

    * Unit/integration tests, stress tests, chaos engineering.

# Critique, limitations, and what’s changed since publication

**Strengths of the book**

* Clear conceptual framing and practical tradeoffs.
* Bridges theory and practice: algorithms explained with operational implications.

# Glossary of key terms

* **ACID** — Atomicity, Consistency, Isolation, Durability.
* **B-tree** — Balanced tree index structure used by many databases.
* **LSM tree** — Log-structured Merge tree, optimized for high write throughput.
* **MVCC** — Multi-Version Concurrency Control.
* **Quorum** — A subset of replicas required to achieve some guarantee.
* **CRDT** — Conflict-free Replicated Data Type.
* **Event sourcing** — Persisting state changes as an append-only sequence of events.
* **Change Data Capture (CDC)** — Streaming modifications from a database (binlog, WAL) into other systems.
* **Idempotency** — Performing an operation multiple times has the same effect as once.
* **Snapshot isolation** — A transactional isolation level where each transaction sees a consistent snapshot.
* **Raft / Paxos** — Consensus algorithms to agree on a value across unreliable processes.
* **Exactly-once** — Strong processing semantics requiring each input to affect the output exactly once.
* **At-least-once / At-most-once** — Weaker processing guarantees for streaming.
* **Materialized view** — Precomputed derived data stored for fast queries.
* **Partitioning (Sharding)** — Splitting a dataset across multiple machines.
